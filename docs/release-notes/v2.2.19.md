# v2.2.19

v2.2.19 是 v2.2.18 之后的维护版本，重点继续围绕任务运行、依赖恢复、订阅拉库、Python 运行环境占用和 Debian 容器兼容性做针对性修复。

## 新增

- 暂无单独对外宣传的新功能，本次主要是问题修复与运行时优化。

## 修复

- **脚本缺依赖自动安装偶发失效**：修复任务执行器在构建脚本运行环境时把 `workDir` 错传成脚本根目录的问题。现在任务运行、缺依赖检测和自动安装重试会优先使用脚本实际所在目录，避免 `PYTHONPATH`、本地模块搜索和自动安装重试环境偏离真实脚本上下文，减少“检测到缺依赖但自动安装不生效”的情况。
- **Node.js 依赖安装报 `EJSONPARSE`**：修复 `data/deps/nodejs/package.json` 损坏后，安装 Node.js 依赖会一直失败的问题。现在安装 / 重装 / 自动补依赖前会先校验 `package.json`，发现 JSON 损坏会自动备份为 `package.json.broken-*`，再根据现有 `node_modules` 重建可用的最小清单，避免 npm 因历史坏文件直接退出。
- **订阅 GitHub 仓库仍全量落盘**：修复订阅管理里已经填写“指定子目录”或“白名单”时，首次 `git clone` 仍会先把仓库根目录和无关文件检出到 `scripts` 的问题。现在有路径过滤时会先进入 sparse checkout，再按规则检出；子目录会改用非 cone 模式，避免 Git 默认保留仓库根文件；白名单也会参与真实检出范围，减少大仓库无关文件落盘。
- **Debian 容器依赖一直卡在“等待重启安装”**：修复 Debian 版在服务重启后恢复 Linux 依赖时，仍走旧的裸 `apt-get update && apt-get install ...` 逻辑，导致与当前依赖管理页的安装策略分叉、状态长期不收敛的问题。现在 Linux 依赖的正常安装和“重启后自动重新安装”会统一复用同一套包管理器策略，Debian / Alpine 的行为保持一致。
- **坏掉的 Python 虚拟环境备份持续堆积**：修复 `deps/python/3.12.broken-时间戳` 这类坏 venv 备份目录长期残留的问题。面板启动时会自动清理旧的 `.broken-*` 备份目录，避免 `deps/python` 越积越大。

## 优化

- **面板启动更轻**：启动时不再默认预热 `3.10 / 3.11 / 3.12` 三个 Python 版本的托管环境。现在会先做历史目录迁移和坏环境清理，真正需要运行脚本、安装依赖或使用对应版本时再按需初始化，降低空闲时的启动负担和文件缓存压力。
- **订阅拉库更省空间**：对带子目录或白名单的 Git 订阅使用 `--no-checkout` + sparse checkout，并在远端支持时启用 `--filter=blob:none`，尽量减少首次拉取时无关 blob 下载和工作区文件落盘。
- **Node.js 依赖安装更稳定**：Node.js 依赖的安装、卸载、强制卸载、备份恢复重装和脚本自动补依赖现在会串行执行，避免多个 npm 进程同时改同一个 `package.json / package-lock.json`，降低清单文件被并发写坏的概率。
- **Python 运行环境目录空间收敛**：保留多版本能力的前提下，减少“默认全量预热 + 保留坏环境备份”造成的额外磁盘占用，降低 `deps/python` 目录膨胀速度。
- **Linux 依赖安装链路统一**：Linux 依赖的安装、重装、重启后恢复现在共享同一套底层包管理器命令构建逻辑，后续 Alpine / Debian / 其它发行版的行为更容易保持一致，也更方便继续维护镜像源、索引刷新和锁冲突处理。

## 升级说明

- 升级到 v2.2.19 后，首次启动会自动清理 `deps/python` 目录下历史遗留的 `.broken-*` 备份环境；如果你仍需要保留这些目录做排查，请先手动备份。
- v2.2.19 起，面板启动阶段不会主动预热三个 Python 版本的托管环境；如果某个版本之前从未用过，首次运行对应 Python 任务或首次安装该版本依赖时，仍会按需初始化。
- Debian 版如果之前有 Linux 依赖长时间停在“等待重启安装”或“安装中”，升级并重启容器后，建议进入依赖管理页观察这些依赖的状态是否已自动收敛；若旧状态仍停留不动，可手动点一次“重装”触发新逻辑。
- 如果某个订阅之前已经把全仓库文件拉进了脚本目录，升级后重新拉取该订阅会按新的子目录 / 白名单规则收敛工作区；清空子目录和白名单后再次拉取，则会自动恢复完整仓库检出。
- 如果之前 Node.js 依赖安装曾出现 `EJSONPARSE / Invalid package.json`，升级后直接重新安装或重装对应依赖即可；原损坏文件会保留为 `data/deps/nodejs/package.json.broken-*` 方便排查。
- 本次更新不涉及数据库结构变更，主要影响任务运行环境、Node.js 依赖清单修复、订阅 Git 拉取工作区、Python 托管目录清理和 Linux 依赖恢复逻辑。
