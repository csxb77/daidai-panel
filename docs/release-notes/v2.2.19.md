# v2.2.19

## 版本概览

`v2.2.19` 是 `v2.2.18` 之后的一次维护与体验并进的更新。  
本次版本主要围绕三条主线持续收口：

- **运行链路更稳**：继续修复任务执行、依赖恢复、订阅拉库、更新判断、备份恢复等高频链路中的边界问题。
- **界面体验更统一**：对 Loading、页面视觉、暗色模式、移动端布局和交互动效进行连续整合，让登录前后界面观感更完整。
- **发布形态更一致**：补齐 Windows 单机版、Docker / 二进制更新路径、Magisk 模块版的运行时与发布一致性，减少“代码已修、实际运行仍不一致”的情况。

## 新增

- **统一 Loading 视觉体系**：面板新增统一的 Loading 展示组件，支持 `infinity`、`spinner`、`dots` 三种动态样式，并可按尺寸、色调和文案形式灵活切换。登录页、依赖管理、订阅管理、按钮加载态、编辑器加载态与局部空白区 Loading 将逐步统一到同一套视觉语言。
- **核心加载态升级为 SVG 动画**：在统一 Loading 组件的基础上，核心加载态进一步切换为更细腻的内联 SVG 动画。全局 `v-loading` 遮罩、按钮 loading、局部页面 loading 的动态表现更加完整，整体更贴近当前版本的界面风格。

## 修复

- **脚本缺依赖自动安装偶发失效**：修复任务执行器构建运行环境时 `workDir` 取值不准确的问题。现在任务运行、缺依赖检测和自动安装重试会优先基于脚本真实目录执行，减少“检测到了缺依赖，但自动安装没有真正生效”的情况。
- **Node.js 依赖安装报 `EJSONPARSE`**：修复 `data/deps/nodejs/package.json` 损坏后，Node.js 依赖安装被直接卡死的问题。现在安装、重装和自动补依赖前会先校验清单；发现 JSON 损坏时，会先备份为 `package.json.broken-*`，再重建可用的最小依赖清单，避免 npm 因历史坏文件直接退出。
- **订阅 GitHub 仓库仍会把无关文件完整拉下**：修复订阅管理中已经指定子目录或白名单时，首次 `git clone` 仍会先把整个仓库检出到脚本目录的问题。现在存在路径过滤时会先进入 sparse checkout，再按规则检出；子目录会使用非 cone 模式，白名单也会参与真实检出范围。
- **Debian 容器依赖长期卡在“等待重启安装”**：修复 Debian 版在服务重启后恢复 Linux 依赖时，仍沿用旧安装逻辑、导致状态长期不收敛的问题。现在 Linux 依赖的正常安装与“重启后自动重新安装”统一复用同一套包管理策略，Debian / Alpine 行为保持一致。
- **坏掉的 Python 虚拟环境备份持续堆积**：修复 `deps/python/*.broken-*` 这类坏环境备份目录长期残留的问题。面板启动时会自动清理旧备份，避免 `deps/python` 目录持续膨胀。
- **脚本管理移动端工作区显示不稳定**：修复脚本管理页在移动端下脚本列表、工作区与编辑器切换不稳定的问题。现在移动端会更明确地区分目录区、内容区和编辑区的切换状态，减少空白区、层级混乱或操作区域突然消失的情况。
- **部分页面仍残留旧主题色元素**：修复仪表盘、开发接口、个人中心、全局增强样式等区域仍残留旧紫色高光、阴影或渐变的问题。本轮已进一步统一为更克制的蓝 / 青 / 绿体系，减少页面局部风格跑偏。
- **后台与暗色模式局部样式不一致**：修复日志页、脚本侧栏、脚本树节点、设置页概览卡片及后台若干卡片在暗色模式下边框、背景、分割线和交互高亮不统一的问题，减少深色环境下的割裂感。
- **仪表盘统计文案易产生误解**：修复仪表盘执行统计区域文案表达不够准确的问题，避免用户把统计含义理解为错误的执行状态或错误的聚合口径。
- **系统健康检查内存状态误报 `0.0%`**：修复健康检查页在已有资源采集能力的机器上，内存项仍显示 `ok / 0.0%` 的问题。现在健康检查会复用真实资源采集结果；如果当前环境确实拿不到内存总量，也会明确显示“资源采集不可用”。
- **Windows 单机版旧 exe 误用导致行为和源码不一致**：修复仓库源码目录里历史遗留本地 exe 容易被误当成正式发布产物的问题。现在 Windows 正式发布链继续统一以 GitHub Release workflow 产物为准，避免出现“源码已修复，但实际运行的还是旧 exe”的情况。
- **备份包仍可能带上异常脚本污染目录**：修复脚本目录中的 `%SystemDrive%` 等异常 / 隔离目录虽然已在页面与恢复链跳过，但备份打包仍可能继续带入的问题。现在备份打包、脚本展示、脚本统计和恢复写回统一复用同一套过滤规则。
- **备份恢复中途失败可能留下半恢复状态**：修复脚本目录、日志目录和 `panel.log` 在恢复时“先清空 live，再逐步复制”的高风险写法。现在会先把恢复结果完整写到同目录 staging，确认成功后再切换 live 目录或文件；如果 staging 失败，现有数据保持原样。
- **系统更新对 Watchtower 空响应兼容不足**：修复 Watchtower 手动触发接口返回 `204 No Content` 或 `200` 空响应体时，面板仍把本次触发误判为失败的问题。现在空响应会被视为“触发成功但无额外 JSON 内容”，不会再因为 `EOF` 直接报错。
- **二进制运行态被残留 Docker 环境变量误导**：修复某些机器虽然实际是裸机二进制运行，但环境变量里残留 `IMAGE_NAME` / `CONTAINER_NAME` 后，面板仍强制走 Docker-only 更新路径的问题。现在只有真正运行在容器内时才会坚持 Docker 更新链，环境变量泄漏不再阻断二进制回退路径。
- **Magisk 模块版 Android 运行时能力链路不闭环**：修复 Magisk 模块版依赖管理里一键安装 Python / Node 后，后端未必能识别“当前就是面具版运行态”，以及容器内任务 PATH 未必能直接发现新解释器的问题。现在模块启动时会显式注入 Magisk 环境标记与运行时目录变量，容器 PATH 也会优先包含对应解释器目录。
- **Magisk 模块升级恢复与交互自检一致性不足**：修复 Magisk 模块升级恢复时隐藏文件（如 `.jwt_secret`）可能未被完整回填的问题；同时修复模块卡片 `action.sh` 与 `scripts/check-runtimes.sh` 的运行时自检 PATH 和真实运行态不一致的问题，减少“面板能跑，但自检却显示缺失”的误导。

## 优化

- **面板启动更轻**：启动阶段不再默认预热 `3.10 / 3.11 / 3.12` 三个 Python 托管环境。现在会优先完成历史目录迁移和坏环境清理，真正需要运行脚本、安装依赖或切换对应版本时再按需初始化，降低空闲状态下的启动负担。
- **订阅拉库更省空间**：对带子目录或白名单的 Git 订阅使用 `--no-checkout` + sparse checkout，并在远端支持时启用 `--filter=blob:none`，尽量减少首次拉取时的无关文件下载与落盘。
- **Node.js 依赖安装更稳定**：Node.js 依赖的安装、卸载、强制卸载、备份恢复重装和脚本自动补依赖会串行执行，避免多个 npm 进程并发修改同一份 `package.json / package-lock.json`。
- **Python 运行环境目录更收敛**：在保留多版本能力的前提下，减少“默认全量预热 + 保留坏环境备份”带来的额外磁盘占用，降低 `deps/python` 目录膨胀速度。
- **Linux 依赖安装链路统一**：Linux 依赖的安装、重装和重启后恢复共享同一套底层包管理器命令构建逻辑，后续维护镜像源、索引刷新与锁冲突处理时更容易保持一致。
- **Loading 与登录后界面观感继续统一**：登录页、依赖管理、订阅管理、编辑器、按钮、局部卡片和全局遮罩的 Loading 风格继续收口；同时对仪表盘、通知页、开发接口、个人中心、日志页和脚本管理等高频页面做了连续多轮视觉精修，页面层次、留白和色彩更统一。
- **页面交互动效更顺滑**：主布局切换、卡片 hover、按钮按压、抽屉 / 弹窗出现、日志区切换、编辑器侧栏展开收起等交互补齐了更克制的过渡效果，减少生硬闪现和各组件各自动的割裂感。
- **暗色模式与移动端体验继续补齐**：主布局、日志页、脚本管理、设置页概览卡片等暗色模式重点区域重新调整了背景层次、边框透明度、阴影、选中态与 hover 态；同时继续优化小屏设备下的按钮排布、工具栏换行与局部点击区域。
- **系统更新路径诊断更可靠**：Watchtower、Docker、二进制三条更新路径的成功态与失败态提示进一步收紧，减少“请求其实已发出，但界面仍提示失败”或“当前明明可回退到二进制更新，却被误挡在 Docker 判断前面”的情况。
- **Magisk 模块发布链与仓库版本更一致**：统一收口 `Magisk/build.sh` 默认版本、`module.prop`、`update.json` 与 Magisk README 中的构建示例，减少本地手工打包、模块卡片提示与仓库当前版本不一致的问题。

## 升级说明

- 升级到 `v2.2.19` 后，首次启动会自动清理 `deps/python` 目录下历史遗留的 `.broken-*` 备份环境；如果你仍需要保留这些目录做排查，请先手动备份。
- 从 `v2.2.19` 开始，面板启动阶段不会再主动预热三个 Python 版本的托管环境；如果某个版本之前从未使用过，首次运行对应 Python 任务或首次安装该版本依赖时，仍会按需初始化。
- Debian 部署如果之前有 Linux 依赖长时间停在“等待重启安装”或“安装中”，升级并重启容器后，建议进入依赖管理页观察状态是否已自动收敛；若旧状态仍停留不动，可手动点一次“重装”触发新逻辑。
- 如果某个订阅之前已经把整个仓库完整拉进脚本目录，升级后建议重新拉取一次该订阅，让工作区按新的子目录 / 白名单规则重新收敛；清空子目录和白名单后再次拉取，则会自动恢复完整仓库检出。
- 如果之前 Node.js 依赖安装曾出现 `EJSONPARSE` 或 `Invalid package.json`，升级后直接重新安装或重装对应依赖即可；原损坏文件会保留为 `data/deps/nodejs/package.json.broken-*` 方便排查。
- 本次版本同时包含一轮前端静态资源更新。如果升级后二级页面仍看到旧的 Loading、旧的配色或旧的动效，建议手动强制刷新浏览器缓存后再观察效果。
- 本次更新不涉及数据库结构变更，主要影响任务运行环境、Node.js 依赖清单修复、订阅 Git 拉取工作区、Python 托管目录清理、Linux 依赖恢复逻辑，以及前端 Loading、UI 和动效展示。
- **Windows 单机版用户**：正式升级请始终以 GitHub Release 中的 `daidai-windows-amd64.zip` 为准；不要继续直接复用仓库源码目录里历史遗留的 `server/*.exe` 做验证或发版。
- **Magisk 模块用户**：本次更新建议重新使用 Release 中最新的模块 ZIP 安装或升级。升级后如果你之前用面板一键安装过 Python / Node，模块卡片自检、交互 shell 和面板任务运行看到的解释器路径会更一致；若仍需手动排查，可继续通过 `action.sh`、`check-runtimes.sh` 或 `ruri ... /bin/bash` 进入容器确认当前 PATH 与解释器路径。
