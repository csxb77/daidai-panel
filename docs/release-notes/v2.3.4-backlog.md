## 2. 终端环境与面板 venv 对不上 + 交互式脚本没法跑

- **状态**：待定
- **现象**：面板里装好的依赖（如 telethon），在 `docker exec` 进容器的终端里 `python3 xxx.py` 报 `ModuleNotFoundError`；交互式脚本（如 Telegram 首次登录要输手机号/验证码）面板任务非交互跑不了，只能进终端手动跑，但环境又对不上。
- **根因**：容器里有三套 Python，面板任务用的是托管 venv，但终端 shell 不继承面板进程的 PATH/PYTHONPATH。
  - 面板装的依赖在 **venv**：`{DATA_DIR}/deps/python/{版本}/`（容器内 `/app/Dumb-Panel/deps/python/3.12/`）的 site-packages。
  - 面板任务能找到依赖，是因为 `docker/entrypoint.sh:104` 只在 daidai-server 进程树里把 venv bin 放进 PATH、并设了 `PYTHONPATH`。
  - `docker exec` 是全新 shell，不继承这套环境，`python3` 落到 `/usr/local/bin/python3`（引导解释器 `/opt/daidai-python/3.12`）或 `/usr/bin/python3`（Alpine 发行版），都不是那个 venv。
  - 在终端里 `pip install` 又会装到第三个位置，越装越乱。
- **临时绕过（无需改代码）**：激活 venv 或直接指向 venv 解释器，零重装：
  - `. /app/Dumb-Panel/deps/python/3.12/bin/activate && python <脚本>`
  - 或 `/app/Dumb-Panel/deps/python/3.12/bin/python <脚本>`
- **建议方案**：
  - **主方案**：给 `ddp` 增加一个在面板环境里跑脚本/开 shell 的子命令（如 `ddp python <脚本>` / `ddp shell`）。`ddp` 已会加载配置+DB，可读「默认 Python 版本」解析出 venv，套上与面板一致的 PATH/PYTHONPATH + 环境变量，在当前 TTY 前台交互执行（stdin 直通，可输手机号/验证码）。
    - 相关代码：`server/cmd/ddp/commands.go`（命令分发）、`server/cmd/ddp/help.go`（帮助）、`server/service/python_runtime.go`（venv 路径解析）、可参考 `server/service/task_executor.go` 的环境构建逻辑。
  - **可选锦上添花**：塞 `/etc/profile.d/daidai.sh`，让 `docker exec -it <容器> bash -l` 登录 shell 自动把 venv bin 加进 PATH。
- **影响**：主方案为后端（Go）+ Dockerfile；可选项为 Dockerfile/entrypoint。

## 3. 容器终端中文乱码 / 编码缺省

- **状态**：待定
- **现象**：终端里中文（文件名、输出）乱码，需手动 `export LANG=C.UTF-8` 写进 `~/.bashrc` 才正常。
- **根因**：镜像未设置 `LANG`/`LC_ALL`，entrypoint 也没导出。
- **建议方案**：Dockerfile 设 `ENV LANG=C.UTF-8 LC_ALL=C.UTF-8`，并在 `docker/entrypoint.sh` 同步导出（双保险）。
  - Alpine 3.22 的 musl（≥1.2.5）原生支持 `C.UTF-8`，无需额外装 locale 包。
  - 一次性解决终端显示 + Python 输出编码，省得每次手动改 `.bashrc`。
  - 相关文件：`Dockerfile`、`Dockerfile.debian`、`docker/entrypoint.sh`。
- **影响**：Dockerfile + entrypoint，纯环境配置。

## 4. 青龙备份导入：定时任务命令丢失 `task` 前缀，变成不可运行的裸脚本

- **状态**：待定（根因待用户提供 1 个数据点最终确认归一化规则）
- **现象**：导入青龙备份包恢复后，定时任务命令缺少 `task`，例如本应 `task 123.py` 变成只有 `123.py`。
- **已验证事实**：面板侧全链路**不剥离 `task`**，是忠实照抄青龙存的命令。
  - 读取：`server/service/backup_qinglong.go:694`（`loadQingLongTasks` 原样读 `Crontabs.command` 列）
  - 落库：`server/service/backup_runtime.go:1001`（`restoreTasks` 原样 `Create`，无改写）
  - 模型：`model.Task` 无 `BeforeSave/BeforeCreate` 钩子；全局 `.Command =` 赋值唯一来源就是上面那行
- **根因**：青龙 `Crontabs.command` 存的不是面板可直接执行的格式，而导入时**未归一化**成面板规范 `task <相对脚本目录的路径>`（参照面板订阅自动建任务：`server/service/subscription.go:1105`）。
  - 面板命令文法（`server/service/script_runner.go:115-124`）要求首 token 必须是 `task` / `desi` / 解释器（`python3`/`node`/...）之一，否则报「不支持的解释器」无法运行。
  - 现有测试样本存的是 `task /ql/scripts/demo.js`（带 `task` 但带绝对路径），用户实际是裸 `123.py`——两种都不符合面板规范。
- **待确认数据点**：取一个出问题任务，看青龙里命令实际是 `task 123.py` / `123.py` / `task /ql/scripts/123.py` 哪种，以定归一化分支。
- **修复方向**：在 `loadQingLongTasks` 给命令加归一化（建议抽成 `normalizeQingLongTaskCommand`）：
  - 首 token 不是 `task`/`desi`/已知解释器 → 前缀补 `task `
  - 剥离青龙路径前缀（`/ql/scripts/`、`/ql/data/scripts/`、开头 `scripts/` 等）→ 转成相对面板脚本目录的路径
  - 补单元测试覆盖三种输入形态
- **影响**：后端单文件改动（`backup_qinglong.go`）+ 测试，不动恢复流程与前端。

## 6. v2.3.3 性能回归：实时日志 `reconnect` 机制 + 前端无限重连导致卡顿

- **状态**：✅ 已实现（后端 `log.go` 流式分支改 30s keepalive 心跳 + else 分支超时改 finished；前端 `LogViewer.vue` 重连熔断+退避；build/vet/handler 测试通过）
- **现象**：升级 v2.3.3 后面板卡顿，典型为定时任务页点「实时日志」后卡顿。
- **回归来源**：`cf6f24b`（v2.3.3）把 `server/handler/log.go` 三处 `done: timeout/finished` 改成 `streamDoneEventForStatus(status)`——只要 DB 任务 `status == running` 就发 `done: reconnect`。改动前是一次性 `done`，前端收到即停。
- **根因链**：
  - 后端 `log.go:114/137/159` + `streamDoneEventForStatus`（`:183`）：信任 DB `status==running` 就发 `reconnect`。
  - 前端 `web/src/views/tasks/components/LogViewer.vue:145-149`：对 `reconnect` **无条件、无退避、无上限** 500ms 重连。
  - 每次重连后端 `tl.ReadAll()` 重发整段历史，前端 `resetLogOutput()` 清空 + `renderedLogHtml` 对整段日志全量 `ansiToHtml`（每 16ms flush 也全量算）→ 接近 O(n²) 重渲染。
- **触发条件**：任务 `status==running` 但**无可流式 TinyLog**：
  1. `conc`/`SuppressLiveOutput` 运行任务（`task_executor.go` 不建 TinyLog）→ else 分支 60s 轮询无果 → `reconnect` → 前端重连 → 持续循环。
  2. 安静长任务（在跑但 60s 无输出）→ 流式分支 60s 空闲 → `reconnect` → 整段历史重发 + 前端全量重渲染。
  - 注：启动 `RecoverAbandonedActiveTasks` 会复位残留 `running`，故崩溃残留不会永久卡；已结束任务状态非 running，发 `finished`（仅 1.5s 老延迟），不受影响。
- **修复方向**：
  - **后端 `log.go`**：`reconnect` 仅在确有 TinyLog 可续流时发。else 分支等满 60s 仍 `tl==nil` → 发 `finished`（该任务无可流式日志）；只有流式分支里 tl 存在、纯安静才 `reconnect`。保留原修复意图同时消除无日志任务死循环。补单测。
  - **前端 `LogViewer.vue`**：① `reconnect` 加次数上限 + 退避（连续 N 次无新数据即停）；② 重连增量续流（不每次清空全量重拉重渲染），或对超大日志做渲染节流/截断。
- **影响**：后端 1 文件 + 测试；前端 1 文件。需回归正常运行任务的实时跟随仍正常。
