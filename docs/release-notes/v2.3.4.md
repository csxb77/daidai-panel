# v2.3.4
<!-- release-title: 终端可用面板环境 · 通知面板名补全 · 青龙导入与全屏修复 -->

## 版本概览

`v2.3.4` 是 `v2.3.3` 之后的一次体验优化 + 问题修复版本，集中处理一批用户反馈的实际问题：容器终端跑脚本环境对不上、v2.3.3「通知面板名称」前端缺失、青龙备份导入丢失命令前缀、脚本页全屏对话框未真正铺满，以及容器终端中文乱码。

本次重点：

- 新增 `ddp python` / `ddp shell`：在容器终端里用与面板任务**完全一致**的 Python 环境（托管 venv + 已装依赖 + 面板环境变量）前台交互跑脚本，解决「面板装的依赖在 `docker exec` 终端 `python3` 找不到」「交互式脚本（如 Telegram 首次登录）没法在面板非交互任务里跑」两个痛点。
- 补全 v2.3.3「通知面板名称」配置项的前端入口（此前只做了后端，界面里看不到也填不了）。
- 修复青龙备份导入后定时任务命令丢失 `task` 前缀（如 `task 123.py` 变成 `123.py`）导致任务无法运行。
- 修复脚本页调试 / 代码运行器「全屏」对话框未真正铺满（顶部露缝、圆角，观感像浮窗）。
- 容器统一 `LANG=C.UTF-8`，解决终端中文文件名 / 输出乱码。
- 修复 v2.3.3 实时日志的性能回归：打开运行中任务的实时日志会持续重连 / 周期性全量重渲染导致卡顿。

本次不涉及数据库结构变更，也不需要用户手动迁移数据。

## 新增

- **`ddp python` / `ddp shell`（容器内用面板环境跑脚本）**：面板把依赖装进托管 venv（`{数据目录}/deps/python/<版本>/`），任务执行时由服务进程注入 venv 的 `PATH`/`PYTHONPATH` 才能找到依赖；但用户 `docker exec` 进来的终端不继承这套环境，直接 `python3 xxx.py` 会落到引导解释器或系统 Python，找不到面板装的包，交互式脚本也没法在面板的非交互任务里跑。新增的 `ddp python <脚本路径> [参数...]` 会用与任务一致的环境（venv 解释器 + 已装依赖 + 面板环境变量）在当前终端**前台交互执行**，`stdin` 直通，可正常输入手机号 / 验证码等；`ddp shell` 则进入一个把 venv 前置到 `PATH` 的交互 shell，里面的 `python3` 即面板解释器。脚本路径相对脚本目录（也可给绝对路径）。

## 修复

- **「通知面板名称」配置项前端缺失**：v2.3.3 已实现后端逻辑（填写后所有通知标题加 `【面板名】` 前缀，多面板共用渠道时区分来源），但系统设置页一直没有对应输入框，导致用户既看不到也填不了。现已在「告警通知」卡片补上「通知面板名称」输入框，可正常填写、保存、回显；留空则不附带，存量用户无感。
- **青龙导入丢失 `task` 前缀导致任务无法运行**：导入青龙备份恢复后，定时任务命令缺少 `task`（例如本应 `task 123.py` 变成只有 `123.py`）。面板命令要求首个词必须是 `task` / `desi` / 解释器，裸脚本名无法执行。此前导入是原样照抄青龙命令、不做归一化。现在导入时统一归一化为面板可执行格式：缺前缀的补 `task `，并剥离青龙脚本根目录前缀（`/ql/scripts/` 等）转成相对脚本目录路径；已合规的命令保持不变。
- **脚本页「全屏」对话框未真正铺满**：脚本管理页的调试 / 代码运行器虽然是全屏对话框，但全局样式把通用对话框的顶部留白（`!important`）和圆角带进了全屏模式，导致它被往下顶出一条缝、四角有圆角，看起来像浮在内容区上的窗口。现已在全屏模式下复位留白与圆角，对话框真正边到边铺满；该修复同时作用于所有全屏对话框。
- **实时日志卡顿（v2.3.3 性能回归）**：v2.3.3 修「日志窗运行中却显示已完成」时改成「只要任务状态是运行中就让前端重连」，而前端对重连无上限、每次还把整段日志重拉并全量重渲染。对没有可流式实时日志的运行任务（并发/抑制输出任务）会陷入持续重连，对安静的长任务则每 60 秒整屏重渲染一次，越跑越卡。现在：安静的运行任务改用轻量心跳维持连接、不再断开重拉（消除周期性全量重渲染）；确无可流式日志的任务直接判完成、不再无限重连；前端也加了重连次数上限与退避兜底。既不再误报「已完成」，也不再卡顿。

## 优化

- **容器终端中文不再乱码**：镜像此前未设置 `LANG`/`LC_ALL`，`docker exec` 进终端时中文文件名 / 输出乱码，需手动 `export LANG=C.UTF-8`。现在 Docker（Alpine / Debian 两套镜像）默认 `LANG=C.UTF-8 LC_ALL=C.UTF-8`，入口脚本同步导出做双保险，终端显示与 Python 输出编码一并正常。

## 升级说明

- **所有部署形态**：本次不涉及数据库结构变更和配置变更，正常替换到 `v2.3.4` 即可。
- **Web 面板用户**：如果升级后仍看到旧界面布局或旧样式（如脚本页全屏对话框、通知面板名称输入框），请先强制刷新浏览器缓存（Ctrl+Shift+R），确保加载到新的前端静态资源。
- **Docker 用户**：请重新拉取 `v2.3.4` 镜像并重建容器（`LANG` / `ddp` 等改动需新镜像生效）；使用 Watchtower 的用户等待自动更新即可。容器内可用 `ddp python <脚本>` / `ddp shell` 在面板环境里跑脚本。
- **二进制 / Windows 单机版用户**：下载 `v2.3.4` 对应压缩包，替换可执行文件即可；请保留原数据目录。`LANG` 编码改动仅作用于 Docker 镜像，二进制部署不受影响。
- **Magisk / KernelSU / APatch 模块用户**：可通过模块管理器更新，也可以下载 `v2.3.4` 模块包手动刷入；数据目录会继续保留。
- **回退建议**：如果升级后遇到异常，建议先备份数据目录，再回退到上一稳定版本 `v2.3.3`。
