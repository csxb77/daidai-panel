# v2.3.7
<!-- release-title: 配置文件编辑、dailycheckin 与运行体验修复 -->

## 版本概览

`v2.3.7` 主要收口上一版发布后用户反馈的几个高频问题：配置文件页面编辑器显示异常且无法输入、`dailycheckin` 这类依赖命令已安装但任务无法执行、Windows 发布包 `start.bat` 换行格式不对、页面切换偶发白屏卡顿，以及旧青龙 Node.js 脚本使用 `require()` 加载新版 ESM-only 依赖时报错。

本次改动重点是“让用户少手动修”：Windows 包自动保证批处理脚本 CRLF，任务命令能直接调用依赖暴露的可执行命令，Node.js 依赖默认安装更偏向兼容 CommonJS 旧脚本。

## 修复

- **修复配置文件页面编辑器显示异常、无法输入**：调整 Monaco 编辑器容器高度继承逻辑，避免普通卡片里初始化时高度被算成 0，导致编辑器只剩一条横线或无法正常输入。
- **修复 `dailycheckin` 脚本无法执行**：任务命令解析现在支持直接运行依赖目录里的可执行命令，例如 `dailycheckin --help`、`task dailycheckin now`，不再只按脚本文件路径查找，避免误报“脚本不存在或命令格式无效”。
- **修复 `python -m <module>` 形式命令不兼容**：支持 `python3 -m dailycheckin --help` 这类模块执行方式，并继续使用面板托管 Python 环境和任务环境变量。
- **修复 Windows 发布包 `start.bat` 换行格式问题**：新增 `.gitattributes` 固定 `.bat/.cmd/.ps1` 使用 CRLF，发布 workflow 在打包 Windows zip 前也会二次转换 `start.bat` 为 CRLF，避免用户双击报错。
- **修复发布前无法自动拦截 `start.bat` LF 的问题**：发布预检脚本会检查 `packaging/windows/start.bat` 是否存在裸 LF，防止同类问题再次进入 Release。
- **修复页面切换偶发卡顿、白屏**：主布局不再使用 `out-in` 等待式切页，改为旧页面短暂淡出、新页面立即进入；同时提高页面缓存上限，减少重复销毁和重建。
- **修复首次切到懒加载页面时等待 chunk 下载导致白屏**：菜单页面支持空闲时预加载，鼠标悬停或聚焦菜单项也会提前加载对应页面。
- **修复部分 Bash 测试环境中 `bash` 命令存在但不可用导致测试失败**：测试会先确认 `bash --version` 可执行，不可用时跳过相关用例。

## Node.js 依赖兼容优化

- **默认安装 CommonJS 兼容版本**：手动安装、缺依赖自动安装、备份恢复重装 Node.js 依赖时，`uuid`、`axios`、`node-fetch`、`got`、`chalk`、`nanoid`、`cheerio`、`https-proxy-agent`、`query-string` 等常见青龙脚本依赖会默认安装兼容 `require()` 的 CommonJS 版本。
- **尊重用户显式版本**：如果用户手动填写 `uuid@9`、`node-fetch@3`、`Git URL`、本地路径等明确来源，面板不会强行改成旧版。
- **未命中映射会明确提示**：安装 Node.js 依赖前会说明是否命中 CommonJS 兼容映射；没命中时会提示“该包未在兼容映射中，将按 npm 默认版本安装。”。
- **增强 `ERR_REQUIRE_ESM` 诊断**：脚本运行命中 `require()` 加载 ESM-only 依赖时，会自动从 `node_modules` 路径或 `require('包名')` 中解析包名，并提示可重装的兼容版本，例如 `uuid@8.3.2`；未命中映射时会提示手动指定兼容 `require()` 的旧版本。

## 升级说明

- Windows 用户升级到本版本后，GitHub Release 里的 `daidai-windows-amd64.zip` 会自动带 CRLF 格式的 `start.bat`，无需再手动用编辑器转换换行。
- 已安装 `dailycheckin` 依赖的用户，可以继续在任务命令里使用 `dailycheckin ...` 或 `python3 -m dailycheckin ...`；如果仍提示找不到命令，请先到依赖管理页确认该依赖安装状态。
- Node.js 依赖页里如果已经安装了 ESM-only 新版依赖，建议卸载后重新安装裸包名，让面板按兼容映射安装旧脚本可用的版本。
