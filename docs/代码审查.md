### 问题清单

#### 严重问题

**问题1：Python 多版本依赖安装存在跨版本误装与误报风险**
- 位置：`server/handler/deps.go:139`、`server/handler/deps.go:209`、`server/service/dependency_auto_install.go:222`、`server/service/dependency_auto_install.go:250`
- 问题：依赖创建会为 Python 3.10、3.11、3.12 同时创建安装任务，但 pip 解析在目标版本 venv/解释器不可用时最终 fallback 到未带版本的 `pip3`。如果服务器未安装 Python 3.10/3.11，3.10/3.11 的安装任务可能实际装进默认 Python，却把数据库记录标成对应版本安装完成。
- 风险：用户后续切换到 Python 3.10/3.11 执行脚本时仍会缺依赖；依赖管理界面还可能显示“已安装”，导致定位困难。这会直接影响本次新增的 Python 版本切换核心功能。
- 建议：请求指定 Python 版本时，不能 fallback 到未验证版本的 `pip3`。应只允许目标版本 venv pip、目标版本 `python -m pip`、`pip3.10/pip3.11/pip3.12`，且执行前校验 `sys.version_info`。目标版本不可用时该版本安装任务应失败并输出“请先安装 Python x.y 或切换版本”。

**问题2：Python 依赖状态/list/export 同样会受 pip fallback 影响**
- 位置：`server/service/dependency_state.go:47`、`server/service/dependency_state.go:68`、`server/handler/deps.go:493`、`server/handler/deps_export.go:95`
- 问题：依赖状态检测、pip 列表和导出在目标版本 venv pip 不存在时，也会走 `NewPipCommandForPythonVersion`，最终可能查询默认 `pip3`。
- 风险：未安装目标 Python 版本时，3.10/3.11 页面、恢复检测、导出依赖可能展示默认 Python 的包，形成错误状态。
- 建议：把 pip 命令解析改为“严格模式”和“允许默认版本模式”。依赖管理按指定版本查询时使用严格模式；不可用时返回明确错误或空结果，不要用默认 pip 代替。

#### 建议修改

**问题3：验证码开启 + 2FA 登录时可能重复使用同一份极验校验结果**
- 位置：`server/handler/auth.go:116`、`server/handler/auth.go:162`、`server/service/auth.go:84`、`web/src/views/login/index.vue:331`、`web/src/views/login/index.vue:350`、`web/src/views/login/index.vue:367`
- 问题：后端先验证极验，再进入账号密码和 TOTP 判断。若账号开启 2FA 且首次未带 TOTP，后端已消费验证码后返回 `two_factor_required`；前端弹出 2FA 输入框后仍保留 `captchaVerified=true` 和旧 `captchaResult`，再次提交时跳过滑块并复用同一份验证码结果。
- 风险：如果极验结果 token 是单次有效或短期强校验，2FA 用户可能在开启验证码后无法完成登录，或遇到“验证码已失效”循环。
- 建议：后端在需要 TOTP 时不要消费验证码，或为 TOTP 二阶段生成服务端短期 captcha pass；简单兜底方案是前端收到 `two_factor_required` 后清空验证码状态，提交 TOTP 前重新滑块。
